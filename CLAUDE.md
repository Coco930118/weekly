## 毎週の実行手順

### タイミング
毎週土曜（または依頼時）に次週分を生成・プッシュする。

### 手順

1. **前々週のデータを削除**（常に最新2週分のみ表示維持）
   - `posts/week_YYYY_MM_DD_YYYY_MM_DD.json`（2週前のファイル）を削除
   - `posts/index.json` の `weeks` 配列から該当エントリを削除

2. **新週のJSONを生成・プッシュ**
   - ファイル名：`posts/week_YYYY_MM_DD_YYYY_MM_DD.json`
   - X投稿：2件/日 × 7日 = 14件
   - Threads投稿：スタンプ発売週は宣伝文4件/日 × 7日 = 28件、通常週は3件/日
   - LINEスタンプ制作週：スタンプサマリー1件を同ファイルに追加

3. **index.json を更新**（weeks配列に新ファイル追加、linestampsは別管理）

4. **main へ PR 作成→マージ**

### 削除ルール例
| 作成する週 | 削除する週 |
|-----------|----------|
| 5/11〜5/17 | 4/21〜4/27 |
| 5/18〜5/24 | 4/28〜5/4  |
| 5/25〜5/31 | 5/5〜5/11  |

### 通常週の投稿構成
- **X**：6:00 🐢しずく / 22:00 🕊ひより（各1日2投稿）
- **Threads**：7:00 🐢しずく / 10:00 🐈‍⬛しらたま / 19:00 🕊ひより（各1日3投稿）
- 合計：35投稿／週

### スタンプ発売週の投稿構成
- **X**：6:00 / 22:00 の2件/日（通常通り）
- **Threads**：7:00 / 10:00 / 14:00 / 19:00 の4件/日（スタンプ宣伝ひとこと形式）
- **LINEスタンプ**：サマリー1件（00:00、platform: "LINEスタンプ"）
- 合計：43投稿／週

### JSONフォーマット（実際の使用形式）
```json
{
  "week": "YYYY-MM-DD_YYYY-MM-DD",
  "posts": [
    {
      "date": "YYYY-MM-DD",
      "platform": "X",
      "time": "06:00",
      "character": "🐢しずく",
      "purpose": "共感獲得",
      "content": "本文（改行は\\n）",
      "quote": "💎 名言"
    }
  ]
}
```

https://coco930118.github.io/weekly/

### platform フィールド一覧
| platform値 | 用途 |
|-----------|------|
| `X` | X（旧Twitter）投稿 |
| `Threads` | Threads投稿 |
| `LINEスタンプ` | スタンプサマリー（フィルターボタン：📦 LINE） |
| `X診断` | X診断コンテンツ |
| `Threads診断` | Threads診断コンテンツ |

**【重要】platform フィールドには日付サフィックスを付けない。**

## 診断コンテンツ（X診断 / Threads診断）運用ルール

### タイミング
- **X診断**：毎週水曜 7:00 に次週7日分（火〜月）を生成・プッシュ
- **Threads診断**：毎週水曜 7:00 に次週7日分を同時生成（X診断と同じ週ファイルに追加）
- 投稿時刻：X診断 12:00 / Threads診断 14:00

### X診断 JSONスキーマ
```json
{
  "date": "YYYY-MM-DD",
  "platform": "X診断",
  "time": "12:00",
  "theme": "テーマ名（カードのサブ表示）",
  "character": "Coco + 🐢しずく",
  "purpose": "保存獲得",
  "content": "4択診断本文",
  "quote": "💎 キーフレーズ",
  "image_prompt": "DALL-E 3英語プロンプト。4分割パネル形式。No text, no letters, no signage.",
  "comment1": "コメント① 回答・解説（各選択肢の解説＋統合コメント）",
  "comment2": "コメント② Coco Methodologyの視点（深掘り・保存補足）"
}
```

### Threads診断 JSONスキーマ
```json
{
  "date": "YYYY-MM-DD",
  "platform": "Threads診断",
  "time": "14:00",
  "theme": "テーマ名（カードのサブ表示）",
  "character": "🐢しずく + 🐈しらたま",
  "content": "4択診断本文",
  "quote": "💎 キーフレーズ",
  "dall_e_prompt": "DALL-E 3英語プロンプト。4パネル形式。No text, no letters, no signage.",
  "reply_1": "返信欄① 解説コメント（各選択肢＋統合メッセージ）"
}
```

**【重要】X診断 と Threads診断 でフィールド名が異なる：**

| 項目 | X診断 | Threads診断 |
|------|-------|-------------|
| テーマ | `theme` | `theme` |
| 目的 | `purpose`（保存獲得など） | 不要（省略可） |
| 画像プロンプト | `image_prompt` | `dall_e_prompt` |
| 解説コメント① | `comment1` | `reply_1` |
| 深掘りコメント② | `comment2` | なし（廃止） |

### Threads診断 コンテンツ方針
- ドメイン：恋愛・家族・人間関係の距離感（Coco Methodology の個人向け領域）
- トーン：慈愛に満ちた哲学者。説教せず、診断を通じて気づきを提供
- 各選択肢の解説で根本心理を解剖し、最後に「距離の設計」視点で統合する
- `reply_1` はThreadsの返信欄①として機能（X診断のcomment2は廃止・comment1に統合済み）

### キャラクター組み合わせローテーション（6パターン必須）
7日間で以下6パターンを割り当てる（いずれか1パターンが2日使用になる）。

| パターン | 組み合わせ | Coco参加 |
|---------|-----------|----------|
| Coco+1匹 | Coco + しらたま or ひより | ○ |
| Coco+2匹 | Coco + しらたま + ひより など | ○ |
| Coco+3匹 | Coco + しずく + しらたま + ひより | ○ |
| 動物1匹 | しずく or しらたま or ひより | ✗ |
| 動物2匹 | しずく+しらたま、しずく+ひより など | ✗ |
| 動物3匹 | しずく + しらたま + ひより | ✗ |

**Cocoのみ単独は禁止。**X診断と同週のThreads診断でキャラ構成が被らないよう調整すること。

### 画像プロンプト（診断共通）
- X診断：4分割（A/B/C/D）パネル構成
- Threads診断：4パネル（選択肢の情景を順に描写）構成
- 必ず末尾に `No text, no letters, no signage.` を付ける
- Cocoが登場する場合は服装の詳細を記述（季節感・素材・色）

### app.js の仕組み（変更時の注意）
- `platform === 'Threads診断'` の完全一致でフィルタリング（日付サフィックス禁止の理由）
- `renderThreadsDiagnosisButtons()` が weekId から週ラベルボタンを自動生成
- `renderThreadsDiagCard()` が `dall_e_prompt` / `reply_1` / `theme` を参照
- `renderXDiagCard()` が `image_prompt` / `comment1` / `comment2` / `theme` を参照
- ブランチは必ず **main** ベースで作成すること（旧feature branchベースはコンフリクトの原因になる）

## Threads 投稿ルール（必須・2026年版）

### 前提
Coco Methodologyの世界観を保ちながら、20代前半〜30代の層にも"自然に自分ごと化される"投稿にする。
「共感と余白」最優先。バズよりも"また読みたくなる空気"を重視する。

### 重要方針
内容のレベルは下げない。「入口の理解コスト」を下げる。
"わかる人に届ける"ではなく、"まだ言語化できていない人に気づかせる"設計にする。

### 禁止事項
- 内容を浅くする
- 説明しすぎる（解説過多）
- 正解を押しつける
- スピリチュアル・自己啓発っぽくする
- 強い断定・ジャッジ
- 「あなた」「みんな」など直接的な呼びかけ

### 構成ルール（必須）

**① 冒頭1行で「静かな共感 or 違和感」**
- 誰でも感じたことがある微細な感情から入る
- 問いかけ or つぶやき形式
- 強すぎない、でも引っかかる一文

**② 中盤で「ズレの気配」をにじませる**
- 説明しすぎない
- 少しずつ"違和感の正体"に触れる

**③ 後半で「構造」を一瞬だけ出す**
- 温度差／距離／認識のズレなど、Coco Methodologyの視点を1文以内で静かに触れる
- 断定せず「そんな気配がある」「ふと気づく」くらいの柔らかさ

**④ 最後は「余白」で終わる（最重要）**
- 答えを閉じない
- 読者が自分の中で考えられる状態にする

### 追加ルール（2026年アルゴリズム対応）
- 最後に「余白を残しつつ、自然に返信したくなる軽い問い」を1行以内で入れる（推奨）
- 一文は15〜20文字以内を強く意識
- 絵文字は世界観を崩さない最小限（温度感を補う程度）

### 質問設計ルール
強制感のない自然な問いにする。思い出や感覚を引き出す形にする。

OK例：「この感覚、最近感じたのはいつ」「どの瞬間に違和感が出た」
NG例：「どう思う？」「みんなはどう？」

### 質問パターンのローテーション（必須）
同じ問いの構造が連続しないよう以下をローテーション。同一パターンは2投稿以上連続禁止。直近7投稿内で2回以上使用禁止。

| パターン | 例 |
|---------|---|
| ① 時系列系 | 最近これを感じたのはいつだった |
| ② シーン想起系 | どの場面でそれが出た |
| ③ 比較・違和感系 | 以前と何か違う感じがあった |
| ④ 内面観察系 | 自分の中で何が動いていた |
| ⑤ 行動振り返り系 | そのとき何をしていた |
| ⑥ 関係性視点系 | 誰かとの間でそれを感じた |
| ⑦ 無意識気づき系 | 気づかないうちにそうしていた |

### 文章スタイル
- 最大10行以内
- 1行1〜2文（短く切る）
- 改行でリズムを作る
- 詩的すぎず、リアル寄り
- 感情は「冒頭の違和感」で最も強く出す

### 言葉のルール
- 抽象ワード（温度差・距離など）は後半に置く
- 最初は具体的な体験・感情
- 専門用語は使いすぎない
- 説明より"気づきの余韻"を優先

### 正しい流れ
感情 → 違和感 → 気配 → 構造 → 余白 → 軽い問い

### NGパターン
- いきなり概念から入る（例：関係の温度差は〜）
- 説明で完結する投稿
- 読み終わって何も残らない

### Threads宣伝文ルール（スタンプ発売週・必須）
- **形式：ひとこと**（1文完結、読んだ瞬間に「欲しい」「これ私だ」と思わせる）
- **量：4件/日 × 7日 = 28パターン**（全28個で重複内容・文言禁止）
- **時間：7:00 / 10:00 / 14:00 / 19:00**
- 抽象表現禁止。読者の悩みや欲求を言い当てた「刺さる一言」にすること

## 運用哲学・執筆プロトコル（永続的指示）

### 1. 運用哲学：『関係の温度と距離を整える技術』
発信は単なる批判ややり方（How）の提示ではなく、在り方（Being）を整えるための「翻訳」。

- **X（組織向け）:** 冷徹な軍師として構造を分析しつつ、経営と現場の「温度」を整える。
- **Threads（個人向け）:** 慈愛に満ちた哲学者として、相手との「距離」を整える。
- **共通のゴール:** 読後感に「破壊」ではなく「調和」と「建設的な解決」を残すこと。

### 2. 執筆プロトコル：『一滴の温かさ』（構成比率・厳守）
1. **【フック/違和感】(30%):** ネガティブな事象や鋭い気づきで目を引く。
2. **【構造分析】(40%):** 「なぜそれが起きるのか」を冷静に言語化する。
3. **【一滴の温かさ】(30%):** 必ず最後に、救い・調和へのヒント・読者の心がふっと軽くなる「温度を整える一滴」で締める。

### 3. 運用・作成ルール
- **トレンド分析:** 直近1週間のトレンドを検索・分析し、上記プロトコルに沿った投稿を作成する。
- **個別依頼の優先:** イレギュラー依頼や特定テーマが指定された場合は、その内容を最優先にしつつ、文体と哲学のみガイドラインを適用する。
- **重複回避:** 過去の投稿・同一スレッド内での表現・結論の焼き直しを厳禁。
- **定型文の回避:** 「〜ではないでしょうか？」等のAI特有の決まり文句を排除し、独自の語彙（仕組み・温度・距離・翻訳等）を使用する。

### 4. 自動修正ルール
- ネガティブすぎる下書きが提示された際は、自動的に「建設的な調和」の方向へ軌道修正して提案する。
- これらの指示と矛盾する案を出さない。
- 明示的に変更依頼がない限り、すべてのセッションで最優先に適用し続ける。

## スケジュール連動・語彙・解像度ルール（永続的追記指示）

### 1. 曜日と内容のリンク（必須）
各曜日のユーザー心理・社会の空気感と内容を必ずリンクさせること。

| 曜日 | 読者の心理・テーマ方針 |
|------|----------------------|
| 月曜 | 週の始まりの重さ。「また動き出す」最初の一歩 |
| 火曜 | 動き始めた後の「なんのため？」が出る日 |
| 水曜 | 週の折り返し。疲れと期待が混在する |
| 木曜 | 「あと少し」の踏ん張り。人間関係の摩擦が出やすい |
| 金曜 | 週末前の解放感と振り返り。距離の整え方を問い直す |
| 土曜 | 自分に戻る時間。孤独・余白・充電のテーマ |
| 日曜 | 静かな問い直し。来週への準備と手放し |

カレンダーの流れを無視したランダム配置を禁止。その日その時の読者の「心の温度」に合わせたトピックを配置する。

### 2. 語彙ルール：「設計」禁止
- **「設計」という言葉の使用を全面禁止。**
- すべて「仕組み」または「仕組み化」に置き換えること。

### 3. 脱・AI抽象表現（解像度の向上）
- 抽象的な表現をすべて禁止。読者が情景や痛みを自分事としてパっと思い浮かべられる具体的な表現に変えること。
- 20年の現場経験に基づいた「手触りのある言葉」に着地させる。
- 概念的な言葉（例：「構造」「在り方」「翻訳」）は使いすぎず、代わりに具体的なシーンや感覚を描写する。

### 4. 読後感の定義：『心のお守りフレーズ』
- Xの `quote`（💎）と Threads の `quote`（💍）は、読者がふとした瞬間に思い出し、心を切り替える「きっかけ」になる短いフレーズにすること。
- 覚えやすく、スクリーンショットに撮りたくなる一言。

### 5. データ管理ルール（毎週の運用）
- 新しい週を作成・反映する際、**前々週分（2週間前）のJSONファイルを必ず削除し、`posts/index.json` からも除去する。**
- 常に最新2週分のみがサイトに表示される状態を維持する。
- 1工程ずつ確認を挟みながら進め、タイムアウトを防ぐ。

- **スタンプ宣伝（platform: `"スタンプ宣伝"`）** は **作成日の翌週火曜〜月曜** のweekファイルに追加（翌週ファイルが未作成の場合は新規作成）
- time: LINEスタンプサマリーは `"00:00"`、宣伝は `"07:00"/"10:00"/"14:00"/"19:00"`
- 既存の X / Threads / X診断 / Threads診断 データは一切削除しない（マージのみ）
- `posts/linestamp_YYYY_MM_DD.json` を新規作成し `posts/index.json` の `linestamps` 配列に追加
- linestampファイルの必須フィールド: `week`（スタンプ発売週のweekId）、`series`、`pattern`、`pattern_description`、`stamps`（24個、各stampは `id`/`character`(配列)/`with_coco`/`type`/`dialogue`/`scene`/`send_timing`/`send_psychology`/`outfit`/`color_scheme`/`image_prompt` を持つ）

**セリフ選定ルール：**
- トレンド分析結果に基づき上記リストから24個を厳選
- **需要が高い順**（よく使われる場面・感情への共感度が高い順）に並べて出力
- リスト外の「今の空気感に合うセリフ」を数点追加提案してもよい
- その週の「中心テーマ」（例：梅雨の気分転換、仕事の励まし、大切な人へ）を設定し、テーマに沿って選定

### トレンド分析フォーマット（STEP 1 出力形式）
1. **【1か月後の季節・風物詩モチーフ】**（小物・装飾・背景に使える要素）
2. **【おすすめの台詞案15選】**（日常で使いやすいものから旬な表現まで）
3. **【推奨されるポーズ・感情】**（キャラクターの動きや表情のアイデア）
4. **【制作のアドバイス】**（ターゲット層・色の使い方のヒント）

テーマはセリフ選定・Cocoの服装・背景小物・販売テキストのトーンに一貫して反映する。

## 画像プロンプト作成規則（2026.05.10更新・永続適用）

### 注意1｜スタイル
やさしい水彩タッチのキャラクターイラスト。ふんわりとした色使い、にじみや筆感を活かした手描き風の質感。整った印象よりも、温かみと柔らかさを優先すること。

**必須フレーズ：**
```
soft watercolor illustration, gentle pastel tones,
hand-painted texture, warm and cozy feeling
```

### 注意2｜キャラクターの目
冷たい・無表情な目はNG。見た人の気持ちが上がるような目の輝きと温かみを表現すること。

**必須フレーズ：**
```
sparkling warm eyes, soft light reflection in eyes,
gentle and cheerful gaze, bright and hopeful expression
```

### 注意3｜文字入れ
手書き風フォントでセリフをイラスト内に自然に配置すること。

**必須フレーズ：**
```
handwritten-style Japanese text『（セリフ）』
naturally placed within the illustration
```

### 注意4｜サイズ・背景
正方形（1:1）、背景完全透過、PNG形式で出力すること。

**必須フレーズ：**
```
1:1 square format, transparent background, PNG,
no background, isolated character
```

### 注意5｜構図の重複排除
他のスタンプと構図・ポーズ・表情が重複しないこと。スタンプごとに異なる視点・角度・ポーズを指定すること。

**変化させるもの例：**
```
front view / side view / looking up / looking down /
full body / bust up / close-up / 3/4 angle
```

### プロンプト出力形式（3ブロック分割）
各スタンプのプロンプトを以下の3ブロックで構成する：

```
[Style] soft watercolor illustration... （スタイル・色味）
[Character] （キャラクター詳細・表情・ポーズ）
[Text/Format] handwritten-style Japanese text『セリフ』... （テキスト・サイズ）
```
