## 投稿作成ルール（毎回必ず適用）

### 永続ガイドライン
`reference/post_guidelines.json` を毎回読み込んで全35投稿に適用する。

### 絶対禁止
- X診断7本は**いかなる場合も変更しない**
- エピソードの捏造・水増し禁止（渡された事実だけ使う）

### 全35投稿の絶対ルール（X・Threads共通）

#### ルール1：既視感ゼロ
- **同じ週の35投稿全体で**、似た表現・似たメッセージ・似た構文を使わない
- 特に以下を週内で重複させない：
  - 「〜と〜は別のことだ」「〜と〜は違う」などの構文
  - 「〜が見ている」「〜を手放す」など同テーマのキーワード
  - note導線の締め方（「〜技術を、noteに書いた」が2本以上出たら表現を変える）
- **ひとこと（quote）はcontentの冒頭文をそのまま使わない**。別角度の言葉で核心を刺す
- **ひとこと（quote）はcontentのテーマ・メッセージと方向が一致していること**（組織語彙とThreadsテーマのミスマッチ禁止）

#### ルール2：フック1行目は「わたしの現場のリアル」を主語にする（X14本）
- X投稿の冒頭1行は、三人称エピソード・ナレーター視点・過去一人称（自分語り）で始めない
- 必ず**「あなた」の現場・状況・感情が主語**となる現在進行形で書く
  - NG：「上の指示通りに動いた。それでも〜」（過去一人称）
  - NG：「現場で叫んだ管理職がいた」（三人称エピソード）
  - OK：「あなたが指示通りに動いても「なぜできないんだ」と言われるなら〜」
  - OK：「あなたは「なぜ敬わないんだ」と言ったことがない。」

## 35投稿完成後の4項目チェック（毎回必須）

35投稿（`posts/week_YYYY-MM-DD_YYYY-MM-DD.json`）を新規作成・修正して完成させるたびに、以下の4項目を自動で再実行する。修正が必要な場合は**提案のみ**行い、勝手に確定しない（Cocoの判断を待つ）。

### ①具体的場面チェック
各投稿が、Xなら板挟みの中間管理職・想いが届かない経営者、Threadsなら恋愛・近すぎる関係に疲れた人、それぞれの読者が自分ごととして場面を思い浮かべられる具体性を持っているか確認する。渡された事実（エピソード本体）にない場面は創作しない。

### ②主語一貫性チェック
- 文章内で主語がブレていないか（同じ語が2つの異なる実体を指すように読めないか）
- X14本：冒頭1行目に「あなた」が明示されているか（機械チェック：1行目に"あなた"の文字列があるか）
- Threads21本：末尾が「あなたは〜？」形式の問いかけで終わっているか（機械チェック：最終行に"？"と"あなた"が両方あるか）

### ③比喩一本化プロトコル
投稿ごとに以下を診断する。
0. プラットフォーム判定（X＝冷徹な軍師・構造で切る／Threads＝慈愛の哲学者・距離で語る）
1. 比喩系統の検出：2系統以上の異なる絵が混在していないか、行番号で列挙
2. 刃の特定：最も強くCocoらしい一行（核心）を特定し、ここは触らない
3. 一本化：刃が乗っている系統を残し、他系統を全消し（引き算のみ、足して整えない）
4. 締めの問いが、削除した比喩に依存していないか確認し、依存していれば刃と同じ語彙で書き直す
5. 導入行（動機・前提）の要否は断定せず、残す/削る両方の効果を示してCocoの判断に返す
6. 締め方判定：note誘導あり＝一歩手前で止める（飢餓感を残す）／note誘導なし＝完結で刺す

出力形式：プラットフォーム判定／note誘導の有無／診断／修正案（全文）／変更点の列挙／判断を返す箇所の明示。

### ④再発パターンチェック（過去に検出した3型）
- **①因果対象の分裂**（th_01型）：比喩の対象が2つの別実体に分裂して因果が飛んでいないか
- **②主語の曖昧さ**（th_18型）：ゼロ代名詞が慣用的に自分を指すため、三人称の話のはずが自分の話に読めてしまわないか
- **③橋文の未回収**（x_03型）：接続文で名指しした答えを、締めでも同じ語彙で回収しているか（「結局何なのか」が最後まで宙に浮かないか）

### 文章のまとめかた：六つの法則（③比喩一本化プロトコルの土台原則）
足して整えない。混ざったら消す。残った一本で刺す。

一．**刃を先に決めて、そこは触らない**
一番強い一行を特定する。直しは全部その周りで起きる。刃を書き換えたら、別の文章になる。

二．**一文章＝一系統**
比喩も語彙も、一本に統一する。同じ行為を二つの言葉で呼ばない。意味は同じでも、読者の中で線が切れる。伝わりにくさの正体は、ほぼこれ。

三．**因果は飛ばさず、動作で繋ぐ**
AだからB、を抽象語で置かない。「攻撃をやめたら」ではなく「その一言をのみ込んだら」。絵を結べる動作にすると、飛びが消える。

四．**頭と尻をブックエンドにする**
一行目に開いたものを、最終行で閉じる。「負けだった」で開けたら「勝ちだった」で閉じる。開きっぱなしの言葉を残さない。

五．**締めの回収先を本文と揃える**
最後の問い、または言い切りが、本文で使った言葉に着地しているか確認する。締めだけ別の語に寄りかかると、そこで線が切れる。

六．**常に引き算**
足して整えない。混ざったら消す。残った一本で刺す。迷ったら、削る方を試す。

順序にも意味がある。一で軸を固定 → 二で系統を絞る → 三で因果を詰める → 四・五で回収 → 六が全工程の姿勢。

## Threads診断（7本）作成後チェック（毎回必須）

`posts/shindan_th_*.json` を新規作成・修正して完成させるたびに、以下の2項目を必ず確認する。確認だけで終わらせず、基準を満たすまで実際に書き直してから保存する。

### ①焦点一致チェック
- 各投稿のframe（フック＋設問）と4択choicesが、同じ1つの軸を指しているか（choicesがフックと無関係な複数テーマに散っていないか）
- reply_1の解説・締めが、その1軸から外れていないか
- 崩れていたら、choices・reply_1・image_promptの該当箇所を書き直す

### ②「舐められず好かれる」明確化チェック
- reply_1の締め（北極星「感情はある。依存はしない。」の前後）で、「どうすれば自分を安売りせず（舐められず）、それでも温かく愛される（好かれる）か」が名言化されているか
- 曖昧・一般論止まりであれば、具体的な一文を追加して明確化する

両チェックとも、ターゲット層（恋愛・夫婦・子育て・友人関係で距離に悩む読者）が読み終えた後に「心が整理された」と感じられる状態を最終基準とする。

## エピソード管理ルール

### ファイル
| ファイル | 役割 |
|----------|------|
| `reference/episodes_soshiki.json` | エピソード本体（E1〜En） |
| `reference/episode_usage_log.json` | 使用履歴（自動更新） |

### 新規投稿生成時の手順

1. `episode_usage_log.json` を読み込む
2. 各エピソードの直近使用日を確認
3. **直近30日以内に使用したエピソードは除外**（使用日リストの最新値が今日-30日より新しければ除外）
4. 残りのエピソードからランダムに選択して各投稿に割り当てる
5. 使用したエピソードIDと投稿日を `episode_usage_log.json` に追記
6. 更新したログファイルを main にプッシュ

### エピソード追加時
- `episodes_soshiki.json` の `episodes` 配列に追記（ID採番: E6, E7...）
- `episode_usage_log.json` に新IDを空配列で追加
- 両ファイルを main にプッシュ

### エピソード分割ルール（自動適用）
「新しいエピソードを預けます」で受け取ったエピソードが複数の原則・思考を含む場合、**確認なしで自動分割して登録する**。
- 1エピソード＝2〜4原則を目安とする（意味的にまとまる単位で切る）
- 分割後のIDは末尾から連番（例：E57追加後はE58, E59…）
- 各分割エピソードに `title` / `theme` / `key_phrases` を付ける
- 分割した全エピソードをまとめて1コミットで main にプッシュ

#### 分割の判断基準（必ず適用）
同一エピソード内の原則が「投稿の核心メッセージとして独立して成立するか」で判断する。

**分割が必要なケース（1投稿に混在すると読者に伝わらない）**
- テーマの主体が異なる：「自分の回復」と「他者への関わり方」
- 行動の方向が逆：「関与しない」と「どう関わるか」
- 概念の領域が別：「戦略的ポジショニング」と「コミュニケーション技術」
- 実話が複数の独立したシーンで構成されている（混在すると投稿が拡散する）

**同一エピソードに収めてよいケース**
- 同じ技術・手法の複数ステップ（例：言葉の設計術4種）
- 同じ状況での連続した判断（例：厳しさを渡しつつ労いも渡す、同日の出来事）
- 同じ「〜はしない」系の並列原則（例：擁護もしない・排除もしない）

**実話エピソードの場合**
- 別々の日・別々の相手・別々のシーンは原則1話ずつ独立させる
- 同じ相手・同じ文脈での一連の出来事は同一エピソードに収めてよい

### 除外エピソードが全て埋まった場合
使用可能なエピソードがゼロになったら、最終使用日が最も古いものから順に再使用する。

## X診断（7本）作成後チェック（ルーティン時・毎回必須）

X診断7本を新規作成・グローアップするたびに、以下の2点を自動で再実行する。修正が必要な場合は**提案のみ**行い、勝手に確定しない（Cocoの判断を待つ）。

### ①焦点の一貫性チェック
- 4択が全て同じ軸（「あなたならどう行動するか」）に揃っているか。他人への感想・評価など異なる軸の選択肢が紛れていないか
- フレーム（フック）の主語・前提と、選択肢の主語・前提が一致しているか（フックで語られていない要素が選択肢に急に出てこないか）
- ターゲット層（板挟みの中間管理職・想いが届かない経営者）が読み終えた後、頭の中が「そういうことだったのか」と整理される着地になっているか

### ②舐められず好かれるチェック
- 正解として提示する選択肢が、「舐められない（芯がある・迎合しない）」と「好かれる（角が立たない・攻撃的でない）」を両方満たしているか
- comment_1でその選択肢が「最も機能する型だ」等の言い切りで、明確に一つだけ選ばれていることが読み取れるか
- 誤答の選択肢が、なぜ機能しないかを感情的な批判ではなく構造的な理由で説明できているか

## X投稿（14本）の書き方ルール（毎回必ず適用）

### コンセプト：「短く・核心を刺して・余韻で誘う」

1. **文字数**: 元の下書きの半分以下に圧縮する
2. **1投稿1メッセージ**: 核心を一つに絞る。詰め込まない
3. **最初の1行**: スクロールを止める力が命。強く、尖らせる
4. **改行**: スマホで読みやすいよう意識的に多め
5. **会話誘導**: 末尾に明示的な質問は入れない。暗に「あなたは？」と思わせる余韻で締める
6. **口調**: 大人の女性らしい、親しみやすく凛とした話し言葉を徹底する
   - **禁止**: 「んです」「んですよね」「だよな」（説明的・男性的・AI的に聞こえる）
   - **推奨語尾**: 「よね」「かな」「だったりする」「そういうもの」「変わってくる」「そうなっていく」「ちがう」
   - 観察として静かに言い切る。解説しない。断定しすぎず、余白を残す
   - 上品なウィットがあり、笑わせようとせず自然に滲み出る

### NG例→OK例
- NG: 「〜なんですよね」「〜ということです」「〜だと思います」（説明・解説口調、AIっぽい）
- NG: 「だよな」（くだけすぎ、雰囲気が合わない）
- NG: 文章が5段落以上、複数メッセージが混在
- OK: 「そういうときは、変わってくる。」「だいたい、そういうもの。」（観察として言い切る）
- OK: 「よね」「かな」で柔らかく余韻を残す
- OK: 5〜7行以内、最後の1文で静かに刺す

## Threads投稿（21本）の書き方ルール（毎回必ず適用）

### コンセプト：「共感→気づき→問いかけ」

1. **質問で終わる**: 全21本、必ず最後に「あなたは〜？」形式の問いかけを入れる（エンゲージメント設計）
2. **口語・会話調**: 「よね」「んだよね」「だよね」を積極的に使う
3. **改行**: 3〜5行ブロック単位で区切る（スマホ縦読み最適化）
4. **具体エピソード**: 1投稿に1つ、読者が自分ごと化できる具体シーンを入れる
5. **文末バリエーション**: 同じ文末が続かないように意識的に変化させる
