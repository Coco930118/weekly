### 勝ちパターン（保存・再読されやすい構造）

**Threads 勝ち投稿（構造的根拠）**
- 「送れなかった言葉がある」「もういいやって思う瞬間がある」「消したメッセージのことをたまに思い出す」
  → 生理的描写が体温を宿している：「心臓がぎゅっとして」「静かに、何かが降りてくる感じ」
  → 言い切らず読者に解釈を委ねる余白がある
  → 「一緒に抱えている感覚（共犯感）」が自然に生まれている
  → 答えを出さず、読者が主人公になれる読後感がある

**X 勝ち投稿（構造的根拠）**
- 「会議の最後3分が静かなチームは」「フィードバックを恐れる部下は弱いんじゃない」
  → 構造的問いが鋭く、マネジメント設計に落とし込まれている
  → 「設計し直す」「場を変える」という能動的視点が「調律師」らしさを出している
  → 現場の空気（会議室・静寂）が一行で喚起されている

### 負けパターン（構造的問題）

**① プラットフォーム混同（最重要課題）**
- X 22:00「距離を置かれた」「温度が見えない関係に疲弊」→ Threads向けテーマがXに混入
- Threads「信頼は積む（現場で20年やってみて）」→ 組織・職場テーマがThreadsに混入
- 混同が起きると両アカウントのキャラが曖昧になり、フォロワーの「期待値」がズレる

**② Threads固定フッターが余白を閉じている**
- 「感情はある。依存はしない。そんな距離の整え方を、ここに残していく💍」が全投稿に固定
- 余韻が最後に「説明」で閉じられ、読者が余白に留まれない
- 保存したくなる「問いの残り方」が消えている

**③ Xの泥臭さ不足**
- 構造指摘は正確だが「現場の温度・違和感・判断の迷い」が入っていない投稿が複数ある
- 参考アカウント群（@take_wagatsuma, @naomicoach7等）の「泥臭い現場感」との差が出ている
- 抽象的な構造指摘に寄りすぎると「正しいがつまらない」になる

**④ Threadsのまとめすぎ**
- 一部投稿が結論を美しく締めすぎて「解説感」になっている
- 「共犯感」ではなく「気づきを与えてもらった感」になっており、読者が受け手になってしまっている

## ■ ver.7.3 改善点（5つ）

### 改善① X/Threads テーマ分離ルールの厳格化（新ルール）
生成後に必ず「このテーマはX検索プールとThreads検索プール、どちらか？」を確認する。
- Xテーマの境界：組織・チーム・マネジメント・構造設計・心理的安全・評価・フィードバック・会議・役割設計
- Threadsテーマの境界：個人の感情・二者間の人間関係・距離感・余白・静かな疲れ・送れなかった言葉・夜の問い
- 判定基準：「主語が組織/チームか、個人/二者間か」で振り分ける。主語が個人でもマネジャーとしての判断・設計はX可。

### 改善② Xに「現場の一行」を必ず入れる（新ルール）
構造指摘の前後に、具体的な現場場面を1行入れる。
- 「月曜の朝、エレベーターで目を合わせなくなったあの日」
- 「水曜の1on1、3分で『特にないです』で終わる関係」
- 「金曜17時の急なミーティング招集」
形式：具体時間帯＋具体場面＋生理的反応（あれば）→ 構造指摘 → 問い

### 改善③ Threads固定フッターの廃止（新ルール）
「感情はある。依存はしない〜」の固定フッターを廃止する。
代わりに、その投稿の文脈に合わせた「余韻の一行」で締める。
- 余韻の条件：言い切らない・解釈を読者に委ねる・問いか観察で終わる・答えを出さない
- 禁止：「〜できる」「〜でいい」「〜しよう」で締める（断言・結論・励まし）

### 改善④ Threadsの「共犯感」を強化する（強化ルール）
「わかる、で終わらない」ルール：
- 「一緒に抱えている感覚」を生む → 「自分だけじゃなかった」ではなく「あなたも、こうだったんじゃないか」の問いかけ
- 答えを出さず一緒に抱えたまま終わる → 解消しない・解決しない・励まさない
- 読者が「これ自分のことだ」と思える余地を、最後の1行まで守る

### 改善⑤ 映像解像度の底上げ（両プラットフォーム）
抽象表現 → 具体表現への変換基準：
- 「何かが」→「胸の真ん中が」「指先が」「視線が」
- 「疲れた」→「指先から力が抜けた」「肩が1センチ落ちた」
- 「感じた」→「心臓がぎゅっとした」「胃のあたりが重くなった」
1投稿に最低1つの身体感覚または具体的場面（時間帯・場所・物）を入れる。

## ■ 痛みの再定義プロトコル

**変換の型**：
- 「〜できない（弱さ）」→「〜しなかった（守った）」
- 「諦めた（失敗）」→「降りた（選んだ）」
- 「逃げた（恥）」→「守った（選択）」
- 「言えなかった（臆病）」→「言葉を選んでいた（丁寧）」

**使用上の注意**：
- 無理な言い換えは読者に届かない。「確かにそう読める」と思える変換のみ使う
- 再定義は1投稿1回まで。全部ポジティブに変換すると嘘になる

## ■ エラー時フォールバック（必須）

分析処理・データ取得・出力生成のいずれかでエラーが発生した場合、以下の優先順位で必ず処理を継続する。

① 実データが取得できない場合
→ 今週生成した35投稿の構造のみを対象に分析を行う（数値は使用しない）

② 分析が途中で停止した場合
→ 完了している範囲から「勝ちパターン」「負けパターン」を抽出し、未分析部分は補完推定で埋める

③ 投稿単位で評価不能な場合
→ 投稿ではなく「構造単位（フック・展開・締め）」で分析する

④ 改善提案が出せない場合
→ 以下の固定改善軸から最低3つ選び提案する
- 痛みの再定義の解像度を上げる
- 映像の具体度を上げる
- 体温（揺らぎ）を足す
- 説明を削り余白を増やす
- 構造と感情の分離を明確にする

⑤ 全体分析が不可能な場合
→ 前週プロトコルとの差分比較のみで改善点を出す

**絶対条件**：処理を中断せず、必ず「改善提案」と「ver更新」まで出力すること。
