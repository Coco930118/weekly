## STEP 3｜痛みの再定義プロトコル

すべての投稿の核心は「痛みの再定義」にある。

**再定義の3段構造：**
1. **認知（痛みをそのまま置く）** — 「〜と感じたことがある」「〜の朝がある」
2. **ずらし（別の解釈を提示する）** — 「でも実はそれは〜だったのかもしれない」
3. **着地（急がない）** — 結論を出しすぎない。問いで終わるか、余韻で終わる。

**再定義の深さチェック（必須）：**
- 表面的な言い換えに留まっていないか（「弱いんじゃない→記憶だ」レベルまで掘れているか）
- 「なぜそう言えるのか」という根拠が映像・場面・身体感覚で支えられているか
- 読者が「これは自分のことだ」と感じられる解像度があるか

## STEP 4｜執筆プロトコル

### 共通原則
1. **映像の喚起を最優先にする** — 抽象概念より具体場面。「心臓がぎゅっとして」「指先が冷えていく」「下書きに保存したまま」
2. **体温は揺らぎで宿す** — 断言しすぎない。「〜かもしれない」「〜だと思ってる」「〜な気がする」
3. **結論を急がない** — 問いで終わるか、余韻で終わる。解説・まとめ・宣言で終わらない。
4. **美しくない本音を一滴足す** — 綺麗すぎる文章には「でも、嫌だった」「正直しんどかった」の一語を入れる余地を探す。

### X専用執筆ルール

**構造：フック（現場場面1行）→ 構造指摘 → 再定義 → 問い（後味の悪さ付き）**

```
[必須] 冒頭1行：具体的な現場場面を置く
例：「先週の月曜、会議が始まる3分前から全員下を向いていた」
例：「フィードバックの後、廊下でその部下が一度も振り返らなかった」

[必須] 構造指摘：なぜそれが起きているかを1〜2行で
[必須] 再定義マーク💎：鋭い言い換えを1行で
[必須] 問い：「〜するか▶」形式 ＋ 判断の迷いや後味の悪さを1語添える
例：「どちらが正しいかはわからない。でも月曜が来る。▶」
```

**X執筆チェックリスト（生成前確認）：**
- [ ] テーマが「組織・職場・チームの構造」であるか（個人感情テーマは除外）
- [ ] 冒頭に現場の具体場面が1行あるか
- [ ] 構造の正しさだけでなく「現場の温度・違和感・迷い」が1行でも乗っているか
- [ ] 問いの後味が綺麗な二項対立で終わっていないか

### Threads専用執筆ルール

**構造：具体場面・身体感覚 → 静かなずらし → 余韻（読者に解釈を委ねて終わる）**

```
[必須] 冒頭：身体感覚か具体場面から始める（「あの1秒」「夜中に既読がついて」「打ちかけて消したやつ」）
[任意] 展開：1〜2回の「ずらし」で静かに別の解釈を提示する
[必須] 結末：読者が自分のこととして引き受けられる1行で終わる
  → 接続詞（「そして」「だから」「つまり」）で終わる行は削除または書き直し
  → 解説・宣言・まとめで終わっている場合は余白に変換する
```

**定型フッター「感情はある。依存はしない。〜💍」の使用ルール：**
- **通常投稿：使わない。** その投稿固有の余韻で終わること。
- **月1〜2本の「まとめ回・ブランド回」のみ使用可。**
- フッターを付けると「宣言文」になり共犯感が失われる。原則削除。

**Threads執筆チェックリスト（生成前確認）：**
- [ ] テーマが「人間関係の個人感情・距離感・余白」であるか（組織構造テーマは除外）
- [ ] 身体感覚または具体場面で始まっているか
- [ ] 最後の行は読者が自分のこととして引き受けられるか
- [ ] 「そして」「だから」「つまり」で終わっていないか（→ 削除または書き直し）
- [ ] 説明過多・まとめすぎ・綺麗すぎになっていないか
- [ ] 共感で終わらず「共犯感（一緒に抱えている感覚）」があるか

## STEP 6｜出力フォーマット（JSON）

週次投稿は以下のJSON形式で `posts/week_YYYY_MM_DD_YYYY_MM_DD.json` に保存する。

```json
{
  "week": "YYYY-MM-DD_YYYY-MM-DD",
  "posts": [
    {
      "date": "YYYY-MM-DD",
      "platform": "X" | "Threads",
      "time": "HH:MM",
      "character": "🐢しずく" | "🕊ひより" | "🐈‍⬛しらたま",
      "purpose": "保存獲得" | "共感獲得" | "返信獲得" | "プロフィール遷移",
      "content": "投稿本文",
      "quote": "💎 or 💍 名言1行"
    }
  ]
}
```

`posts/index.json` に週ファイル名を追記することで GitHub Pages の自動反映が動く。

## 自動グローアップ実行モード（土曜13時）

今週生成した35投稿すべてを分析し、ver.N+1を出力する。

**分析観点：**
1. 「保存獲得した勝ちパターン」と「反応が鈍かった負けパターン」を抽出
2. 痛みの再定義の深さ
3. 映像の喚起の解像度（具体場面＋生理的描写）
4. 体温の宿し方（揺らぎのある語尾、美しくない本音の一滴）
5. XとThreadsの魅せ方の分離ができているか

**参考アカウント照合（プラットフォーム別に必ず分離して確認）：**
- X側：@naomicoach7、@take_wagatsuma、@kento329_GMO_TE、@dx_saru、@Cotackey、@mottajapan の最新投稿傾向と比較
- Threads側：「Threads 距離感 共感 バズ 上位投稿」傾向と比較

**出力：**
1. 分析サマリー（勝ちパターン・負けパターン）
2. 改善提案3〜5点（わたしらしさを崩さない範囲）
3. 改善点を反映した完全プロトコル全文（ver.N+1）

## エラー時フォールバック（必須）

分析処理・データ取得・出力生成のいずれかでエラーが発生した場合、以下の優先順位で処理を継続すること。

① 実データが取得できない場合
→ 今週生成した35投稿の構造のみを対象に分析を行う（数値は使用しない）

② 分析が途中で停止した場合
→ 完了している範囲から「勝ちパターン」「負けパターン」を抽出し、未分析部分は補完推定で埋める

③ 投稿単位で評価不能な場合
→ 投稿ではなく「構造単位（フック・展開・締め）」で分析する

④ 改善提案が出せない場合
→ 以下の固定改善軸から最低3つ選び提案する
- 痛みの再定義の解像度を上げる
- 映像の具体度を上げる
- 体温（揺らぎ）を足す
- 説明を削り余白を増やす
- 構造と感情の分離を明確にする

⑤ 全体分析が不可能な場合
→ 前週プロトコルとの差分比較のみで改善点を出す

**絶対条件：処理を中断せず、必ず「改善提案」と「ver更新」まで出力すること。**
